# Sodiq School — Diagnostic Result Field Map

This document classifies every field rendered by the existing `/client` UI so the
backend, admin panel, and API know **what is computed vs. entered vs. fixed**.

Field types:

- **AUTO** — generated by the backend (UUIDs, codes, hashes, timestamps).
- **DERIVED** — computed by `client/src/lib/compute.js` from per-question labels.
- **MANUAL** — entered by an admin; varies per student/result.
- **STATIC** — fixed UI strings/branding (not stored per result).
- **CONFIG** — set on Exam / global settings (grade thresholds, grading bands).
- **PLACEHOLDER** — currently hardcoded in client pages but clearly intended to
  be real data (cohort rank, percentile, male-rank, radar values). Marked
  separately because they must be either populated from real data or hidden.

## 3. Per-question subject data (MANUAL labels → DERIVED everything)

Each subject (Math, English, Critical Thinking) carries ~25 questions. Labels
are entered by the assessor; ALL aggregate numbers are computed.

| Field       | Allowed values                                       | Notes                                     |
| ----------- | ---------------------------------------------------- | ----------------------------------------- |
| `id`        | `"Q1"…"Qn"`                                          | Display id.                                |
| `marks`     | integer ≥ 0                                          | Maximum marks for the question.           |
| `difficulty`| `Oson` / `O'rta` / `Qiyin`                           | Weight 1/2/3.                              |
| `strand`    | free text (curriculum strand)                        | Group axis.                                |
| `topic`     | free text                                            | Group axis.                                |
| `subTopic`  | free text                                            | Display only.                              |
| `skill`     | free text                                            | Radar / cross-skill grouping.              |
| `bloom`     | `Eslab qolish` / `Tushunish` / `Qo'llash` / `Tahlil` / `Baholash` / `Yaratish` | Cognitive level.                  |
| `reasoning` | `Deduktiv` / `Induktiv` / `Analitik` / `Fazoviy` / null | Reasoning type.                       |
| `gradeLevel`| e.g. `"5-sinf"`                                       | For gap zones.                             |
| `framework` | free text (e.g. `"TIMSS: Number"`)                   | Display only.                              |

| `earned`    | integer ≥ 0, ≤ marks                                  | Earned marks.                              |

| `evidence`  | short free-text sentence                              | Per-question narrative.                    |

### DERIVED (per subject) — see `lib/compute.js`

| Field                        | Formula                                                                              |
| ---------------------------- | ------------------------------------------------------------------------------------ |
| `rawScore`                   | `Σ earned`                                                                           |
| `percent`                    | `round(rawScore / totalMarks * 100)`                                                 |
| `correctCount` / `partialCount` | counts                                                                            |
| `ci` (low/high/margin)       | `margin = round(√(p·(100-p)/n) / 2)`                                                 |
| `band` (juda/yaxshi/ortacha/rivoj/sayoz) | thresholds 90/80/70/60                                                   |
| `tiers[Oson/O'rta/Qiyin]`    | `{n, correct, wrong, pct, weight}`                                                   |
| `hardTierPct`                | `tiers.Qiyin.pct`                                                                    |
| `kdi`                        | `Σ correct·weight / Σ n·weight × 100`                                                |
| `mastery`                    | from KDI: 95/85/70/55/40 thresholds                                                  |
| `technicalLost` / `gapLost`  | sum of `marks - earned` for matching errorType                                       |
| `adjusted`                   | `rawScore + technicalLost`                                                           |
| `potential`                  | `round((adjusted + kdi + hardTierPct) / 3)`                                          |
| `byStrand` / `byTopic` / `byBloom` / `byReasoning` / `byGradeLevel` / `bySkill` | grouped `{n, correct, percent, lowConfidence}` |
| `weakestTopic` / `strongTopics` | from sorted `byTopic` (n ≥ 2)                                                     |
| `gapZones` (below/at/above)  | below `<55`; above `≥70 & (avgWeight≥2.3 or analysisFrac≥0.6)`; else `at`            |
| `overallRisk`                | `kdi ≥ 70 & below ≤ 3 → PAST`, `kdi ≥ 55 → O'RTA`, else `YUQORI`                     |
| `riskItems`                  | from `gapZones.below[0..2]` + technical-error row if `technicalLost > 0`             |
| `growthForecast`             | `[Hozir, 3oy, 6oy, 12oy] = percent + headroom × {0, 0.45, 0.7, 0.95}`                |
| `errorRoster` / `technicalErrors` / `gapErrors` | wrong questions enriched with "harder same-skill solved" check                |

Rule (CLAUDE.md): groups with `n < 3` → qualitative label + low-confidence
flag, NOT a precise number.

## 5. Admission threshold (CONFIG)

Source: `resource/result.text`.

Per grade, **per subject %** is the threshold for that subject's pass mark. Per
CLAUDE.md correction at the bottom of `result.text`:

> Har bir fan kesimidagi qabul qilinsin ko'rsatgichi kerak emas. U faqat
> umumiyda tursin.

→ The "qabul qilinsin" verdict is only shown on the **composite/summary**, not
per subject. The threshold per grade is the **minimum subject percent that
counts toward admission** (used in the composite gate).

| Grade | Matematika | Tanqidiy fikrlash | Ingliz tili |
| ----- | ---------- | ----------------- | ----------- |
| 5     | 30%        | 40%               | 30%         |
| 6     | 30%        | 40%               | 30%         |
| 7     | 35%        | 30%               | 35%         |
| 8     | 35%        | 30%               | 35%         |
| 9     | 35%        | 25%               | 40%         |
| 10    | 35%        | 25%               | 40%         |
| 11    | 35%        | 25%               | 40%         |

Store as `Exam.admissionThresholds: JSON` keyed by grade × subject. Admin
editable. The composite verdict uses these thresholds + composite potential.

## Open questions (decisions surfaced — see `docs/implementation-notes.md`)

1. **Per-subject threshold vs. composite gate.** `result.text` lists per-grade
   per-subject percents AND says the verdict only belongs in the composite.
   Interpretation: thresholds gate the composite; per-subject is reported as a
   number but no "qabul qilinsin" badge per subject. Admin should be able to
   toggle the gate rule (any-fail-blocks vs. composite-only).

2. **Placeholder cohort numbers.** No real cohort yet. Two options: (a) show
   "no cohort yet" if `realData.percentile == null`, or (b) compute rank from
   published results in the same exam+grade. Recommendation: implement (b) and
   render "—" when published peer count `< 5`.

3. **Manual narrative blocks.** §13/§14/§15 are currently boilerplate Astro
   markup. Either keep them as fully-templated text (no per-student variation),
   or surface them as admin-editable rich text. The reasonable middle ground:
   store an optional override on `Result.manualContent.{outlook, parent,
   committee}`; if null, fall back to the current template.

4. **English / Critical-Thinking computation.** Both use the same per-question
   structure as Math (verified via `data/english.json`, `critical-thinking.json`
   and a single shared `computeReport`). Same field map applies. Only the
   `subject` label and per-question content differ.

5. **Bloom & skill placeholders.** `BLOOM_LEVELS[].ph` and `skillRadar[]`
   carry hardcoded fallback %s when the test doesn't measure that level.
   Decision: surface them as Admin overrides under `manualContent` with a
   "fallback when no measurement" note. Backend never computes these.
